开发规范

代码规范

  • 项目名调整，不要直接使用sms

  • 开发人员不允许新增服务/模块；新增模块/服务由技术经理操作；

  • 项目包名：com.公司域名.项目名缩写;

  • 命名规则：需要符合java命名约定，并且不允许出现英文单词拼写错误。

  • 项目结构：整个项目组，统一项目结构和项目风格。

    • controller 必须
    • service 必须
      • –impl 必须
    • mapper 必须
    • config
    • dto 必须 pojo包不允许使用domain 命名（本项目不采用领域模型）
    • entity 必须
    • remote 必须
    • utils
    • filter
    • interceptor
    • annotation
    • aspect
    • handler
    • enums
    • constants

  • dto和dao
    所有字段必须写中文注释

  • 统一返回：因采用了ruoyi的框架，则统一按照ruoyi的规范

    • feign：使用R返回
    • 普通controller：使用AjaxResult返回

  • 关于实体注入：所有注入，不允许使用Autowire，使用构造方法注入。

  • 服务间调用，请使用DTO接收接口返回，不允许使用entity接收。

  • 服务间调用，如果涉及到事务，一个业务逻辑尽量不要写超过2个以上的服务调用，避免出现分布式事务。

  • 涉及外部接口调用的，必须记录接口调用实例数据到数据库表，可以写个通用拦截处理：

    • 消费方：
      • 请求时间，请求头，请求参数，请求体，返回时间，返回body
    • 提供方：
      • 请求ID，请求时间，请求头，请求来源，请求参数，请求体，返回时间，返回body
        如果是同步接口，需要记录同步时间，同步参数，以及同步条数

        配置文件规范

  • 对于所有服务相同的配置，使用nacos共享配置的方式，引用配置。

  • 如service-a.yml,redis.yml:

  • redis.yml

    spring:
  redis:
    host: 127.0.0.1
    port: 6379
    database: 0
    timeout: 5000
    password: 123456
    jedis:
      pool:
        maxActive: 3
        maxIdle: 3
        minIdle: 1
        maxWait: -1

    service-a.yml

    spring:
  application:
    # 服务名
    name: service-a
  cloud:
    nacos:
      config:
        # 是否开启配置中心 默认true
        enabled: true
        # 配置中心地址
        server-addr: localhost:8848
        # 配置文件后缀
        file-extension: yml
        # 配置对应的分组
        group: PEARL_GROUP
        # 命名空间 常用场景之一是不同环境的配置的区分隔离，例如开发测试环境和生产环境的资源（如配置、服务）隔离等
        namespace: ba42e722-81aa-48f1-9944-9dca57d5f396
        # Nacos 认证用户
        username: nacos
        # Nacos 认证密码
        password: 123456
        # 支持多个共享 Data Id 的配置，优先级小于extension-configs,自定义 Data Id 配置 属性是个集合，内部由 Config POJO 组成。Config 有 3 个属性，分别是 dataId, group 以及 refresh
        shared-configs[0]:
          data-id: redis.yml
          group: PEARL_GROUP
          refresh: false

  • 自定义属性前缀为项目名

  • 不允许服务启动指定自定义配置文件

    数据库规范

  • 开发人员不允许修改表结构，如需要修改，请提交项目/技术经理操作；由项目/技术经理修改项目表结构。

  • 表结构修改，不允许直接修改数据库，必须使用SQL脚本进行修改，存放在项目resouces/sql目录:

    服务名_init.sql                 初始化SQL
服务名_patch_20220606.sql       修改SQL

分支模型图

分支规范

  • 开发人员只能从develop分支拉取代码；
  • 代码提交合并develop前，必须先进行更新代码（git pull），对于有冲突的文件，必须要进行对比，确认素有修改都是自己修改的，然后在进行提交，防止代码回溯（即别人的代码被覆盖）；否则，将被拒绝合并。
  • 代码出现冲突时，必须 要与冲突代码提交者进行确认冲突内容，双方确认无误方可处理；未解决冲突将被拒绝合并。
  • 代码提交develop时，必须描述清楚做了什么，否则将被拒绝合并。
    • 提交动作有add、update、remove、bugfix
    • 提交格式为[动作]+[操作描述信息]，例如：git commit –m “[update] 将员工精确查询修改为模糊查询”

      分支发布

  • 开发测试发布：
    • develop分支，来源于feature分支的合并，不允许直接提交到develop分支。手动发布到开发环境；
    • release分支，来源于develop分支的合并，不允许从其它任何分支合并，不允许直接提交代码到release。手动发布到测试环境；
  • 生产环境发布：
    • 不允许从其它任何分支合并，不允许直接提交代码到master，不允许直接从master分支发布。
    • 首先测试环境中对release分支进行测试，测试通过了，再合并到master，然后对master打tag，tag名称为发布的版本号。
    • jenkins手动触发生产环境发布（基于tag发布）。

项目制度

周例会

  • 每周安排1次例会，周一早上或周五下班前，例会内容为：上周任务情况，本周任务安排。

任务划分

  • 任务划分，必须具体到每天任务。
  • 原则上，开发成员必须按照例会安排好的任务，完成当天任务，每天完成任务后提交Pull Request合并代码。
  • 到下班时间仍未完成当天任务晚上需要加班完成；
  • 如果加班后扔不能按时完成，需要汇报项目经理，已完成进度，未完成内容，预期完成时间。
  • 当天未完成的任务，请确保提交的代码不会影响开发环境运行。
